## Q. 왜 퇴사했다가 다시 커머스웨어로 재입사했나?

## Q. 재입사할 만큼 좋았다면서 왜 또 이직하려고 하나?

**짧은 버전(구두 답변용)**:
> "사람·문화는 만족하지만, 처우와 성장 기회 면에서는 지금보다 더 나아갈 곳이 있다고 판단했습니다."

- 주의: "성장/커리어 천장"을 구체화해달라고 하면 바로 답할 수 있어야 함 — 예: 지금 역할에서 더 이상 맡을 수 없는 것(시스템 설계 주도권, 더 큰 트래픽/규모, 신기술 도입 결정권 등)을 1~2개 예시로 준비해두기

**틀 3 — 신기술 도입 결정권**

**짧은 버전(구두 답변용)**:

- 후속 질문 대비: "그럼 AI 없이도 이 코드 짤 수 있어요?" → "네, 시간이 더 걸릴 뿐 로직은 제가 이해하고 있습니다" 식으로 자신감 있게. 실제로 설명 못 하는 부분이 있으면 인터뷰 전에 코드 다시 훑어보고 채워둘 것.

## CS 기본기 — 설명 답변 초안 (심화 + 꼬리질문 대비)

`study-plan.md` 섹션 2 체크리스트에 대응. 1차 답변만으로 끝나지 않게, 실제 면접에서 나올 법한 꼬리질문과 답변 방향까지 같이 정리. 본인 언어로 소리 내어 말하는 연습은 필수 — 이 문서는 암기용이 아니라 뼈대.

### 네트워크
- **TCP 3-way handshake / TCP vs UDP**: TCP는 SYN → SYN-ACK → ACK 3단계로 연결을 맺음. 2-way가 아니라 3-way인 이유는 양쪽 다 "내가 보낸 SYN을 상대가 받았다"는 걸 확인해야 초기 시퀀스 번호를 서로 안전하게 합의할 수 있기 때문(2-way면 클라이언트가 자기 SYN이 상대에게 도달했는지 확인할 방법이 없음). UDP는 연결 설정 없이 바로 데이터를 보내 빠르지만 순서·도달을 보장 안 함.
  - 꼬리질문 1. "그럼 연결 종료는 왜 4-way인가요?" → TCP는 전이중(양방향) 통신이라 FIN도 양쪽이 각자 보내야 함. 서버가 클라이언트의 FIN을 받아도 자기 쪽에서 보낼 데이터가 남아있을 수 있어 바로 FIN을 못 보내고, 그 사이 ACK만 먼저 보낸 뒤 나중에 자기 FIN을 보냄(half-close).
  - 꼬리질문 2. "TIME_WAIT 상태는 왜 필요한가요?" → 마지막 ACK가 유실되면 상대가 FIN을 재전송할 수 있어 이를 받아 재응답하기 위해, 그리고 같은 포트로 새 연결이 생겼을 때 이전 연결의 지연 패킷이 섞여 들어오는 걸 막기 위해 일정 시간 소켓을 완전히 닫지 않고 대기.
  - 꼬리질문 3. "HTTP/1.1과 HTTP/2 차이는?" → HTTP/1.1은 커넥션당 요청을 순차 처리(Head-of-Line 블로킹 이슈로 브라우저가 커넥션을 여러 개 열어 우회), HTTP/2는 하나의 커넥션에서 여러 요청/응답을 멀티플렉싱하고 헤더를 압축(HPACK)해 오버헤드를 줄임.
- **HTTP/HTTPS, TLS handshake 개요**: TLS handshake에서 인증서 교환 후 비대칭키로 대칭키(세션키)를 안전하게 교환하고, 이후 실제 데이터는 빠른 대칭키로 암복호화. 비대칭키를 계속 안 쓰는 이유는 연산 비용이 커서 매 요청마다 쓰면 성능이 크게 떨어지기 때문.
  - 꼬리질문. "TLS 1.3이 1.2보다 빠른 이유는?" → handshake 왕복 횟수를 2-RTT에서 1-RTT로 줄였고(불필요한 협상 단계 제거), 취약한 암호 스위트를 제거해 협상 과정 자체를 단순화했기 때문.
- **REST 원칙, 상태코드 4xx/5xx 구분**: REST는 자원을 URI로, 행위를 HTTP 메서드로 표현하고 서버가 클라이언트 상태를 유지하지 않는(무상태) 것이 핵심. 4xx는 요청 자체 문제(잘못된 파라미터·인증 실패), 5xx는 서버 처리 실패.
  - 꼬리질문. "PUT과 PATCH 차이, POST와 PUT의 멱등성 차이는?" → PUT은 리소스 전체를 교체(같은 요청 여러 번 보내도 결과 동일 = 멱등), PATCH는 일부만 수정. POST는 매번 새 리소스를 생성할 수 있어 멱등하지 않음 — 재시도 로직을 짤 때 이 멱등성 여부가 "재시도해도 안전한가"를 가르는 실질적 기준이 된다는 점까지 엮어서 설명하면 실무 이해도가 드러남.

### OS
- **프로세스 vs 스레드, 컨텍스트 스위칭**: 프로세스는 독립된 메모리 공간, 스레드는 프로세스 내 메모리(힙/코드/데이터)를 공유. 컨텍스트 스위칭은 CPU가 실행 흐름을 바꿀 때 레지스터·스택 상태를 저장·복원하는 과정 — 스레드 전환이 프로세스 전환보다 싼 이유는 메모리 매핑을 새로 안 해도 되기 때문.
  - 꼬리질문. "그럼 스레드가 많다고 항상 빠른가요?" → 아니다. 코어 수보다 스레드가 많아지면 컨텍스트 스위칭 오버헤드가 늘고, CPU-bound 작업은 코어 수에 맞춘 스레드 수가 유리하며 I/O-bound 작업만 스레드를 늘리는 게 이득 — Tomcat이나 커스텀 스레드풀 사이즈를 정할 때 이 기준(CPU-bound vs I/O-bound)으로 판단한 경험이 있는지 (확인 필요) 점검.
- **동기/비동기, 블로킹/논블로킹**: 블로킹/논블로킹은 제어권을 붙잡고 있는지 여부, 동기/비동기는 결과를 호출자가 직접 기다려 처리하는지 콜백/이벤트로 나중에 받는지 — 서로 독립된 축.
  - 꼬리질문. "Spring MVC는 기본적으로 어느 조합인가요?" → 기본은 스레드 하나가 요청 하나를 블로킹으로 처리(동기+블로킹). WebFlux 같은 리액티브 스택으로 가면 논블로킹+비동기로 바뀌는데, 왜 지금 프로젝트가 굳이 리액티브를 안 썼는지(트래픽 규모 대비 필요성 낮음, 팀 러닝커브 등) 설명할 수 있으면 좋음.
- **데드락 발생 조건 4가지**: 상호배제·점유대기·비선점·순환대기 — 하나만 깨도 예방 가능.
  - 꼬리질문. "실제로 데드락을 겪어본 적 있나요?" → DB 트랜잭션에서 두 트랜잭션이 테이블 A→B, B→A 순서로 락을 거는 경우가 전형적인 실무 데드락 케이스. 락 획득 순서를 통일(항상 A→B)해서 순환대기를 끊는 방식으로 해결한다는 논리를 실제 겪은 사례(확인 필요)로 채울 것.

### DB
- **인덱스 동작 원리(B+Tree), 역효과 나는 경우**: 리프 노드에 정렬된 데이터를 둬서 범위검색·정렬에 강함. 역효과: 낮은 카디널리티 컬럼, 쓰기 잦은 테이블의 과도한 인덱스(INSERT/UPDATE마다 인덱스도 재정렬), 인덱스 컬럼에 함수/형변환을 걸어 인덱스를 못 타는 경우.
  - 꼬리질문. "복합 인덱스는 컬럼 순서가 왜 중요한가요?" → B+Tree는 왼쪽 컬럼부터 정렬되므로 선두 컬럼이 조건절에 없으면 인덱스를 못 탐(leftmost prefix 규칙) — 실제 쿼리 패턴(어떤 컬럼으로 자주 필터링하는지)에 맞춰 순서를 정한 경험이 있으면 예시로.
- **트랜잭션 격리수준 4단계 + 문제**: Read Uncommitted(더티리드) → Read Committed(반복읽기 불일치) → Repeatable Read(팬텀리드 가능, MySQL은 갭락으로 상당수 방지) → Serializable(동시성 최저).
  - 꼬리질문. "MVCC가 뭔가요?" → 읽기 작업이 락을 안 걸고도 일관된 스냅샷을 보게 해주는 방식(버전 관리로 구현) — Repeatable Read에서 반복읽기 정합성을 락 없이 어느 정도 보장해주는 핵심 메커니즘. "그럼 왜 팬텀리드는 못 막나요?" → 스냅샷은 기존 행의 버전은 관리하지만 새로 INSERT된 행까지는 못 막아서(MySQL은 갭락으로 이 부분을 보완).
- **락 종류(Shared/Exclusive, 낙관적/비관적)**: 비관적 락은 충돌이 잦다고 가정해 미리 락(SELECT FOR UPDATE), 낙관적 락은 충돌이 드물다고 가정해 버전 컬럼으로 커밋 시점에만 체크.
  - 꼬리질문. "낙관적 락 충돌이 나면 어떻게 처리하나요?" → 보통 예외를 던지고 재시도 로직(재조회 후 재적용)을 붙이거나, 사용자에게 "다시 시도해달라"는 응답을 줌 — 재시도 횟수 제한을 안 두면 무한 루프 위험이 있다는 점까지.
  - **Fitness In Service 결제/세션 도메인 연결**: 결제 상태 갱신처럼 동시 요청 시 정합성이 중요한 곳에 비관적 락/버전 체크 중 무엇을 썼는지, 왜 그걸 골랐는지(트래픽 양·충돌 빈도 예상) 실제 구현 기준으로 채울 것 (확인 필요).

### 자료구조/알고리즘
- **시간복잡도 Big-O**: O(1)/O(log n)/O(n)/O(n log n)/O(n²) — 문제 풀 때마다 왜 이 복잡도인지 말로 설명하는 연습 필수.
  - 꼬리질문. "공간복잡도는 왜 같이 얘기하나요?" → 시간을 줄이려고 메모이제이션/해시맵 같은 추가 공간을 쓰는 트레이드오프가 흔해서, "이 풀이는 시간을 O(n²)→O(n)로 줄이는 대신 공간을 O(n) 더 씁니다" 식으로 트레이드오프를 짚어야 완성된 답변.
- **HashMap 내부 동작(해시충돌 처리)**: 해시값으로 버킷 위치를 정하고 충돌 시 체이닝(일정 개수 넘으면 트리 전환), 로드팩터(0.75) 초과 시 리사이징.
  - 꼬리질문. "멀티스레드 환경에서 HashMap 쓰면 안 되는 이유는?" → 리사이징 중 연결리스트 구조가 꼬이면서 무한루프/데이터 유실 위험(Java 7 이하에서 특히 유명한 이슈) — ConcurrentHashMap은 버킷 단위로 락을 쪼개거나(Java 8+는 CAS+synchronized 블록 단위) 락 경합을 최소화해서 이 문제를 해결. "그럼 HashMap 대신 항상 ConcurrentHashMap 쓰면 되나요?" → 락/CAS 오버헤드가 있으니 단일 스레드 컨텍스트에선 굳이 안 씀 — 상황에 맞는 선택이 핵심.

## Spring/Java 심화 — 답변 초안 (심화 + 꼬리질문 대비)

`study-plan.md` 섹션 3 대응. `(확인 필요)` 표시된 부분은 실제 Fitness In Service/커머스웨어 경험으로 본인이 채워야 완성됨 — 이 프로젝트 자료엔 내부 구현 디테일 정보가 없어 임의로 채우지 않음. 채워 넣을수록 답변 설득력이 커지므로 이 항목이 가장 우선순위 높음.

- **MyBatis vs JPA**: MyBatis는 SQL을 직접 제어할 수 있어 복잡한 조인·튜닝에 유리, JPA는 매핑 자동화로 생산성이 높지만 N+1 등 함정이 있음. "왜 MyBatis를 선택/유지했나" → **레거시 시스템을 JPA로 전면 전환하는 건 리스크 대비 이득이 크지 않다고 판단해서 MyBatis를 유지**. 신규 프로젝트라면 JPA도 검토 가능하지만, 이미 안정적으로 돌아가는 레거시 쿼리 자산을 굳이 바꿔서 얻는 이득보다 전환 과정에서 생기는 리스크(회귀 버그, 검증 비용)가 더 크다는 게 판단 기준.
  - 꼬리질문 1. "그럼 JPA를 써야 한다면 N+1 문제는 어떻게 해결하나요?" → fetch join으로 연관 엔티티를 한 쿼리에 같이 가져오거나, `@BatchSize`/`default_batch_fetch_size`로 IN 절 배치 조회, 또는 애초에 지연로딩이 필요 없는 화면이면 DTO 직접 조회(QueryDSL 등)로 우회.
  - 꼬리질문 2. "MyBatis는 영속성 컨텍스트가 없는데 동일 트랜잭션 내 캐싱은 어떻게 하나요?" → MyBatis 1차 캐시(세션 단위)가 있긴 하지만 JPA 영속성 컨텍스트만큼 자동 관리는 안 됨 — 트랜잭션 내에서 같은 데이터를 여러 번 조회해야 하면 애플리케이션 레벨에서 직접 캐싱하거나 조회 결과를 재사용하도록 설계해야 한다는 차이를 짚을 것.
- **트랜잭션 전파(Propagation)**: 기본값 REQUIRED는 기존 트랜잭션에 참여, REQUIRES_NEW는 독립된 새 트랜잭션으로 부모와 무관하게 커밋/롤백. **실무에서 REQUIRES_NEW를 의도적으로 적용해본 사례는 아직 없음** — 개념과 용도(부모 트랜잭션 롤백과 무관하게 남아야 하는 로그/이력 저장 등)는 알고 있다고 정직하게 답하되, "필요성을 느낀 상황이 아직 없었다"는 식으로 답변. 면접에서 나오면 개념 설명 + 언제 쓰는지(예시)까지는 대답 가능하게 준비.
  - 꼬리질문. "`@Transactional`이 같은 클래스 내부에서 self-invocation(자기 호출)할 때 왜 안 걸리나요?" → Spring AOP는 기본적으로 프록시 기반이라 외부에서 프록시 객체를 거쳐 호출해야 트랜잭션 어드바이스가 적용됨. 같은 클래스 내부 메서드가 `this.method()`로 자기 자신을 호출하면 프록시를 거치지 않아 어드바이스가 무시됨 — 이 함정에 실제로 걸려본 경험이 있으면 강력한 답변 소재.
- **Spring Batch**: Job은 최상위 단위, Step은 청크(read-process-write) 또는 Tasklet 기반 개별 단계. `SessionUpdateTasklet`처럼 왜 청크가 아니라 Tasklet으로 짰는지 → **청크 방식은 대용량 데이터 처리에 맞는 구조라 오히려 번거롭고, 빅데이터 수준이 아닌 일반적인 배치 작업이라면 Tasklet이 더 단순하고 적합하다고 판단**. "데이터 규모에 맞는 방식을 골랐다"는 논리로 설명.
  - 꼬리질문. "배치가 중간에 실패하면 어떻게 재시작하나요?" → JobRepository가 Job/Step 실행 이력(상태·파라미터)을 메타테이블에 저장해두기 때문에, 같은 JobParameters로 재실행하면 실패한 Step부터 재시작 가능(청크 방식이면 마지막 커밋된 청크 이후부터). 멱등하게 설계돼 있는지(재실행해도 중복 처리 안 되는지)까지 확인해서 답할 것 (확인 필요).
- **JWT — Access/Refresh 전략**: Access는 짧게(BO 1시간), Refresh는 길게(MO 7일) — BO/MO를 다르게 둔 이유는 **① BO는 내부 관리자용이라 보안 민감도가 높아 짧게, ② MO는 일반 사용자용이라 자주 재로그인시키면 불편하니 편의성을 우선해 길게, 두 가지가 함께 작용**. 보안 하나만이 아니라 "관리자 페이지 vs 일반 사용자 앱"이라는 사용 맥락 차이가 근거라는 점까지 설명하면 설득력 있음.
  - 꼬리질문. "Refresh 토큰이 탈취되면 어떻게 대응하나요?" → Refresh 토큰 rotation(사용할 때마다 새 토큰 발급 + 이전 토큰 폐기)으로 재사용 탐지, 또는 서버 측에 토큰 저장소(Redis 등)를 두고 로그아웃/이상탐지 시 강제 무효화(블랙리스트) — 현재 구조가 stateless(서버 저장 없음)인지 stateful인지에 따라 답이 달라지므로 (확인 필요) 실제 구조 파악 후 답변.
- **Redis — 캐싱/세션**: 세션 저장소로 쓰면 여러 인스턴스가 세션 공유 가능(무상태 서버 확장). 캐싱 전략 — **면접 전 실제 코드에서 "캐시 미스 시 DB 조회 후 캐시에 채우는 방식(Cache-aside)이 맞는지", "TTL을 데이터 종류별로 몇 초/분으로 잡았는지"를 직접 확인해둘 것 (아직 미확인)**. 대부분의 Redis 캐싱은 Cache-aside가 기본값이라 그대로일 가능성 높지만, 자신 있게 답하려면 실제 설정값(TTL 숫자)까지 한 번 코드에서 확인 권장.
  - 꼬리질문. "캐시와 DB 데이터가 어긋나면(정합성 깨지면) 어떻게 하나요?" → 쓰기 시점에 캐시를 함께 무효화/갱신(write-through 또는 write-invalidate)하는 게 1차 방어, 그래도 놓치는 경우를 대비해 TTL을 짧게 잡아 "최악의 경우에도 TTL 지나면 자동 정정"되도록 안전망을 이중으로 두는 설계 원칙을 설명.
- **멀티모듈 구조(core/bo-api/mo-api/batch)**: 공통 로직을 core에, BO/MO/배치를 독립 모듈로 분리해 의존성·빌드/배포 단위 분리. 실제 이점 → **AWS EC2 위에서 Docker 기반으로 서비스 운영 중이며, 모듈별로 독립 배포가 가능해 무중단 배포 구성이 됨** — 배치 모듈 배포/재시작이 API 서버(BO/MO) 가용성에 영향을 주지 않는다는 게 핵심 이점.
  - 꼬리질문. "모듈 간 순환 의존이 생기면 어떻게 막나요?" → core는 다른 모듈을 몰라야 한다는 규칙(단방향 의존)을 빌드 설정(Gradle 모듈 의존성 선언)으로 강제 — 이 규칙이 실제로 깨질 뻔했던 사례나, 신규 기능 추가 시 "이건 core에 둘까 bo-api에 둘까" 판단 기준(여러 모듈에서 재사용되는지 여부)이 있으면 예시로 준비 (확인 필요).

## 시스템 설계 — 화이트보드 연습 틀 (심화 + 꼬리질문 대비)

`study-plan.md` 섹션 4 대응. Fitness In Service 기준 "트래픽 10배 시나리오" 연습용 틀. 대기업 면접에서는 1~4단계 설명 후 아래 꼬리질문으로 파고드는 게 일반적이므로, 답만 외우지 말고 "왜 그 대안을 선택했는지" 비교 논리를 준비할 것.

**1단계 — 현재 구조 그리기**: 클라이언트(BO/MO) → API 서버(bo-api/mo-api) → DB, 배치(batch)가 별도로 DB 접근하는 그림을 먼저 그린다. Redis(세션/캐시)도 표시.

**2단계 — 병목 예상 지점 짚기**:
- API 서버: 요청 폭증 시 단일 인스턴스면 CPU/커넥션 풀 고갈 → 무상태 설계 전제로 수평 확장(L4/L7 로드밸런서 뒤에 인스턴스 추가) 논의
- DB: 읽기 트래픽이 늘면 레플리케이션(읽기 전용 복제본으로 분산), 쓰기 트래픽 자체가 늘면 샤딩(데이터를 여러 DB로 분할) 검토 — 두 개는 다른 문제라는 점을 구분해서 설명
- 캐시: Redis 하나로 부족해지면 캐시 클러스터링, 캐시 히트율 낮은 데이터는 애초에 캐싱 대상에서 제외
- 배치: 트래픽 급증 시점과 배치 실행 시점이 겹치면 DB 부하 경합 → 배치 시간대 분리 또는 배치도 읽기 복제본 활용

**3단계 — 캐싱 전략 구체화**: Cache-aside(조회 시 캐시 미스면 DB 조회 후 캐시에 채움) 기준으로, TTL을 자주 안 바뀌는 데이터는 길게, 실시간성 중요한 데이터는 짧게 또는 이벤트 기반 무효화로 설정.

**4단계 — 메시지 큐 도입 판단**: 동기 호출(예: 알림 발송)이 트래픽 증가로 API 응답 지연을 유발하면, 큐에 던지고 비동기로 처리하는 전환 시점과 트레이드오프(즉시성 손실 vs 응답속도 개선)를 설명.

### 꼬리질문 대비 — 각 단계에서 파고들 만한 질문

- **"샤딩키는 뭘로 잡을 건가요?"** → 자주 함께 조회되는 데이터가 같은 샤드에 있어야 조인/트랜잭션이 덜 꼬임(예: user_id 기준 샤딩이면 "특정 유저의 모든 데이터"는 한 샤드에 몰림). 특정 값에 트래픽이 몰리는 핫스팟을 피해야 한다는 점도 언급 — 예를 들어 가입일시처럼 순차 증가하는 값을 그대로 키로 쓰면 최신 데이터에만 쓰기가 몰릴 수 있음.
- **"레플리케이션은 지연(replication lag)이 있는데, 방금 쓴 데이터를 바로 읽어야 하면 어떻게 하나요?"** → 결제 직후 확인처럼 즉시 일관성이 필요한 read는 master로 보내고, 목록 조회처럼 약간의 지연이 허용되는 read만 replica로 분산하는 식으로 read 종류에 따라 라우팅을 구분. 이걸 "read-after-write consistency" 문제라고 부른다는 용어까지 알아두면 좋음.
- **"캐시와 DB 정합성이 깨지면 어떻게 하나요?"** → 쓰기 시점에 캐시를 무효화(또는 갱신)하는 게 1차 방어, TTL을 안전망으로 병행 — "완벽한 실시간 정합성"보다 "최종적으로는 맞는" 수준(eventual consistency)을 목표로 잡는 게 현실적이라는 트레이드오프 인식을 보여줄 것.
- **"메시지 큐를 쓰면 메시지가 유실되거나 중복 처리될 수 있는데?"** → 유실 방지는 큐 자체의 ack/재시도 메커니즘(컨슈머가 처리 완료 후에만 ack), 중복 처리 방지는 멱등키(idempotency key)를 둬서 같은 메시지가 두 번 와도 결과가 같도록 설계 — "at-least-once 전달을 전제로 컨슈머를 멱등하게 짜는 게 일반적"이라는 원칙을 짚을 것.
- **"Redis가 다운되면 서비스는 어떻게 되나요?"** → 캐시 용도면 캐시 미스로 취급해 DB로 폴백(응답은 느려지지만 서비스는 유지), 세션 저장소로만 쓰고 있다면 로그인 세션이 끊기는 영향까지 인지하고 있어야 함 — "단일 장애점(SPOF)"이 어디인지 스스로 짚어내는 태도가 중요. Redis 이중화(Sentinel/Cluster) 필요성까지 언급하면 심화 답변.
- **"트래픽이 10배가 아니라 순간적으로 100배 튀면(스파이크) 어떻게 하나요?"** → 오토스케일링은 반응 속도에 한계가 있으므로, API Gateway/로드밸런서 단에서 rate limiting이나 circuit breaker로 과부하 시 일부 요청을 거절해 전체 시스템 다운을 막는 방어적 설계(graceful degradation) 개념까지 언급.

**연습 방법**: 주말마다 이 4단계 + 꼬리질문을 종이/화이트보드에 그리면서 소리 내어 설명 — 혼자 하기 어려우면 스터디메이트나 ChatGPT/Claude를 면접관 삼아 질문받는 롤플레이도 효과적. 답을 대사처럼 외우지 말고 "왜 대안 A 대신 B를 골랐나"의 비교 논리 자체를 체화할 것.
