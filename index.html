Design

설계 원칙

DDD 기반 멀티모듈

공유 도메인은 Core, 관리자(BO)·회원(MO)은 별도 모듈. 공유 코드는 반드시 Core에 두어 경계를 강제.

계층 & 경계 분리

Controller → Service → Mapper → Entity. DTO는 API 계층, VO는 Service↔Mapper — 도메인이 DTO에 의존하지 않게.

응답·예외 통일

모든 응답은 ApiResponse<T>, 예외는 BusinessException+에러코드 enum으로 일관되게 처리.

정합성·추적성 우선

상수 중앙화로 매직리터럴 제거, TraceId 필터+AOP로 요청 전 구간 로깅해 장애를 추적 가능하게.

Backend Design

백엔드 설계 포인트 — 배경 · 설계 · 이유

주어진 정책을 백엔드에서 정합성 있게 풀어낸 설계입니다.

상품 · 멤버십 · 수업 3계층 — 도메인/DB 설계

배경

회원이 실제 소유하는 건 상품이 아니라 차감·예약의 기준이 되는 권리(멤버십)이고, 개인+그룹이 섞인 패키지 상품도 팔아야 했습니다.

설계

판매 단위(Product)·소유/차감 단위(Membership)·운영 단위(Class)를 엔티티로 분리해 스키마에 반영(상품→멤버십 1:N, 멤버십→수업 N:1), 결제 성공 시 상품을 unpack해 멤버십을 N개 생성하도록 구현.

이유

패키지 상품을 단품 로직 변경 없이 확장할 수 있고, 화면별 표기 소스(스케줄=수업명, 보유함=멤버십명)를 분리해 혼선을 없앴습니다.

수업 취소·폐강의 차감 복구 정합성

배경

취소·폐강은 멤버십 차감 복구·정원·강사 급여에 연쇄로 영향을 주고, 취소 권한이 수업 유형·마감기한에 따라 달라지는 정책이 있었습니다.

설계

차감 복구는 트랜잭션으로 즉시 반영, 폐강은 배치 스케줄러가 최소 수강 인원 미달 시 자동 취소 → 전원 예약 취소 + 차감 복구 + 알림. 권한 정책은 수업 유형·마감 전후로 백엔드에서 분기 처리.

이유

돈·자원에 직결되는 상태(차감·정원)는 정합성을 최우선으로 두고, 권한은 운영 실수를 구조로 막도록 분기했습니다.

Problem Solving

문제 → 원인 → 해결 → 결과

운영하며 실제로 부딪힌 문제들. 코드 레벨 상세는 면접에서 설명 가능합니다.

알림·이력이 트랜잭션에 잘못 묶이는 문제 · 트랜잭션·배치

문제

알림 발송이나 실패 이력이 본 트랜잭션에 묶이면, 롤백 시 이력이 사라지거나 커밋 전에 알림이 먼저 나갔습니다. 배치에서는 한 건 오류가 전체를 롤백시켰습니다.

원인

부가 작업(알림·이력)과 핵심 처리가 같은 트랜잭션 경계 안에 있었음.

해결

실패 이력·주문 만료 배치는 REQUIRES_NEW로 건별 독립 트랜잭션, 알림은 @Async+커밋 후 콜백(afterCommit)으로 "커밋 성공 이후에만" 발송.

결과

부분 실패가 전체를 되돌리지 않고, 롤백된 거래의 알림이 잘못 나가는 일이 사라짐.

도메인 구분 한계 → 패키지 구조로 확장 · 확장성·운영안정

문제

수업을 class_type으로 도메인만 구분해, 같은 도메인 내 개인/그룹/듀엣 형식을 나눌 수 없었고 개인+그룹 혼합 패키지도 팔 수 없었습니다.

원인

판매(상품)·운영(수업) 단위가 한 축(class_type)에 묶여 있어 확장이 막힘.

해결

product_type='PACKAGE'+구성항목 테이블을 신설해 결제 시 unpack하여 멤버십 N개 생성. 기존 상품 목록 쿼리는 status·display 조건이라 패키지가 자동 포함 → 구매 플로우만 분기하고 나머지는 무변경.

결과

단품 로직에 영향 없이 패키지 판매 확장. "운영 중 자산은 건드리지 않고 확장은 격리"라는 원칙과 일치.

코드만 통과시키고 넘어가면 코딩테스트 감은 늘어도 면접 설명력은 안 늠 — 풀고 나서 아래 4단계를 소리 내어 말하는 것까지 한 세트.

  1. 문제 재진술 — "이 문제는 결국 OO를 OO 조건 안에서 찾는 문제입니다"
  2. 접근 후보 비교 — 브루트포스부터 언급하고 최적화 근거 설명
  3. 선택한 자료구조/알고리즘과 이유 — "그래서 OO를 써서 조회를 O(1)로 줄였습니다"
  4. 시간/공간복잡도 정리 — 트레이드오프까지 언급

"하고잡이" — CJ그룹 인재상
2025~2026년 채용에서 "선언에 그치지 않고 결과로 증명해내는 사람(We Who Prove It)"로 재정의됨. CJ올리브네트웍스는 이 인재상을 AX(AI Transformation) 채용과 직접 연결("AX 혁신 주도할 '하고잡이' 찾습니다"). 다만 그룹 전체 채용 브랜딩 용어라 CJ올리브네트웍스 자체 채용 페이지엔 등장하지 않음 — 자소서·면접에서 1회만 자연스럽게 인용, 이후는 "결과로 증명"으로 풀어쓸 것.

DX와 AX — CJ올리브네트웍스 맥락

사용자가 제공한 예상질문 18개를 기존 리서치 질문과 대조 — 중복 6개(자기소개/지원동기·강점단점·팀갈등·최근IT관심사·실패프로젝트·배포경험)는 이미 다른 섹션에 있어 제외. 아래 12개만 신규.

Q. 입사 후 목표는 무엇인가요?

입사 후 1년 안에는 CJ ONE의 회원·인증 도메인을 실무 코드 레벨까지 완전히 파악하고, 갭으로 인정한 Kubernetes 운영 경험을 실무에서 실제로 쌓는 걸 첫 목표로 삼고 있습니다. 이후엔 더현대·투홈 통합에서처럼 정책 변경·장애 상황에서 영향 범위를 빠르게 파악하고 대응하는 역할까지 올라가고, 장기적으로는 회원 시스템 설계 결정에 직접 참여하는 포지션으로 성장하고 싶습니다.

짧은 버전: 1년 내 CJ ONE 도메인 완전 파악 + K8s 실무 경험 확보, 이후 정책 변경·장애 대응을 주도할 수 있는 역할로 성장. · 꼬리질문 "목표 달성 여부 확인법?" → K8s 실무 클러스터 운영 여부, 도움 없이 장애 원인 파악 가능 여부로 자가 점검.

Q. 실수를 했을 때 어떻게 대응했나요?

실수를 발견하면 숨기지 않고 바로 알리는 게 우선입니다. (확인 필요: 실제 겪은 실수 사례) 발견 즉시 영향 범위를 파악해 팀에 공유하고, 되돌릴 수 있는 부분은 롤백·핫픽스로 대응했습니다. 이후엔 재발 방지를 위해 (확인 필요: 체크리스트/테스트 보강 등)로 프로세스를 조정했습니다. 실수 자체보다 얼마나 빨리 알리고 얼마나 정직하게 영향 범위를 밝히는지가 신뢰를 좌우한다고 생각합니다.

⚠️ 본인 실제 사례로 채워야 완성 — 없으면 "실패했거나 아쉬웠던 프로젝트" 소재 재사용 가능. 꼬리질문 "숨기고 싶었던 적은?" → 늦게 알려질수록 영향 범위가 커진다는 논리로 정직 축 유지.

Q. 본인이 팀에 기여할 수 있는 점은 무엇인가요?

두 가지입니다. 첫째는 변경의 영향 범위를 끝까지 추적하는 꼼꼼함 — 바우처 정책 변경 대응처럼 전수 조사 후 배포하는 습관이 팀 전체 배포 안정성에 도움됩니다. 둘째는 모르는 것을 숨기지 않는 태도 — K8s 경험 없음을 그대로 밝힌 것처럼, 팀 내에서도 모르는 부분을 빨리 드러내야 서로 도울 수 있다고 믿습니다.

짧은 버전: 영향 범위를 끝까지 추적하는 꼼꼼함 + 모르는 걸 숨기지 않는 정직한 커뮤니케이션. ⚠️ "이 직무에 필요한 역량" 답변과 구조 유사 — 연달아 나오면 표현 조정 필요.

Q. 지원 직무와 전공·경험이 다르면 어떻게 적응할 것인가요? 참고용

이 문구 그대로보다 "기술 스택 갭"(K8s·Kotlin)으로 변형되어 나올 가능성이 큼. 가장 가까운 기존 경험(Docker 배포·운영)에서 원리를 확장해 개념을 빠르게 학습하고, 실무 투입 후 검증하며 채워나가는 방식으로 적응한다고 답변.

꼬리질문 "그래도 적응 안 되면?" → 숨기지 않고 팀에 도움 요청 — 정직 축과 통일.

Q. 팀보다 개인의 성과가 우선이라는 상황에서 어떻게 행동하겠는가?

개인 성과가 팀 전체 결과에 어떤 영향을 주는지부터 따집니다. 기간계 시스템 고도화에서 AA 역할을 맡았을 때도 개인 구현 속도보다 팀 전체가 같은 기준으로 개발할 수 있도록 공통 모듈 설계·교육에 더 시간을 썼는데, 결과적으로 그게 더 큰 기여였습니다. 개인 성과를 무시하진 않지만 판단 기준은 항상 "팀 전체 결과에 어떤 영향을 주는가"입니다.

꼬리질문 "개인 평가에서 손해 봐도 괜찮나?" → 팀 결과가 좋아지면 그 안에서 기여도 같이 드러난다는 논리로 방어.

Q. 지원 직무 관련 현재 개선이 필요하다고 생각하는 기술·역량은?

Kubernetes 실무 운영 역량이 가장 먼저입니다 — Docker 배포·운영 경험은 있지만 K8s는 개념 학습 단계라, Pod/Service/Deployment부터 학습 중이고 입사 후 실무 클러스터로 채우고 싶습니다. 두 번째는 MSA·분산 시스템 설계 — 지금까지는 멀티모듈 모놀리식까지만 다뤄봤습니다.

⚠️ K8s 갭 답변과 소재 겹침 — 같이 나오면 MSA 비중을 더 실어 반복처럼 안 들리게 조정.

Q. CJ 기업 3가지 특성(정직/열정/창의) 중 본인과 가장 잘 맞는 것은?

"정직"이 답변축 — K8s 경험 없다고 있는 그대로 말씀드린 것처럼, 모르는 것과 아는 것을 정확히 구분해 말하는 태도. "하고잡이/We Who Prove It"(선언에 그치지 않고 결과로 증명)와 자연스럽게 연결 — "정직이 전제, 그 위에서 하고잡이는 끝까지 실행해 결과로 증명하는 것"으로 위계 연결. "하고잡이" 단어는 1회만 언급.

Q. (예상) DX·AX가 본인 직무와 어떻게 연결된다고 생각하나요?

"CJ ONE 슈퍼앱 진화"를 대규모 회원 데이터의 AI 초개인화 방향으로 이해, 그 전제가 되는 "안정적 데이터 적재·정합성"이 본인 역할이라는 답변. 회사가 DX/AX를 명시적으로 구분 정의한 문서는 없으므로 단정적으로 말하지 말 것.

Q. 무리한 일정의 지시가 내려온다면?

"무조건 맞춘다"도 "무조건 반발한다"도 아닌, 영향 범위를 먼저 전수 조사해 "여기까지는 되고 여기는 리스크가 있다"를 구체적으로 공유하고 우선순위를 같이 정하는 방식 — 바우처 정책 변경 대응 사례 재사용.

Q. 앞으로 커리어 방향 / 개발문화 형성 방안 / 인생관은?

커리어: 안정성·정합성 중심 백엔드 도메인 심화 + AI 에이전트 활용 결합. 개발문화: "모르는 걸 숨기지 않는 문화"가 최우선. 인생관은 본인 실제 가치관으로 채워야 하는 미완성 항목.

Q. 본인이 개발한 것 중 CJ ONE에 적용 가능한 게 뭐가 있나요?

고객자산(예치금/H.Point/바우처) API 경험 + Fitness In Service의 JWT Access/Refresh·Redis 세션 설계 경험을 CJ ONE의 회원·포인트·인증 도메인에 1:1로 매칭.

Q. CJ ONE의 최근 사업 방향 / 최근 관심 있게 본 뉴스는?

이마트24·야놀자 등 비계열 제휴 확대("개방형 멤버십")와 CJ PAY 우리통장(우리은행 제휴 핀테크)을 최신 동향으로 언급 — 제휴사가 늘수록 회원 인증·계정 관리가 단순 로그인을 넘어 여러 파트너사가 신뢰하는 인프라가 돼야 한다는 논리로 지원 이유와 연결.

Q. (예상) 회원서비스 직무를 맡으면 구체적으로 어떤 업무를 하게 될 것 같나요?

회원가입/탈퇴/재가입, 소셜로그인 연동, 휴면계정 전환, 개인정보 동의관리, 세션·토큰 인증체계 + 제휴사 계정·포인트 매핑까지 추정. 확정 정보 아님 — "제 나름대로 추정" 톤 유지.

Q. CJ ONE이 Kotlin/Ktor 쓰는 팀도 있다던데 아시나요? / Kotlin 써보셨나요?

실무 경험 없음을 정직히 인정 + JVM 기반이라 Java 경험으로 빠르게 전환 가능하다는 논리(K8s 답변과 동일 톤). Ktor는 Spring보다 가볍고 코루틴 기반 비동기 중심이라는 것만 개념적으로 파악.

Q. 고객사 요구사항 vs 회사 기술력, 하나 고른다면? / 무리한 고객사 요구사항 해결법은?

무조건 한쪽을 고르기보다, 요구사항을 그대로 수용했을 때의 리스크를 구체적으로 설명해 고객사가 판단할 정보를 주고 안전한 대안을 제시 — 더현대·투홈 바우처 정책 변경 대응이 실제 사례.

Q. 서버 메모리 튜닝 과정은? / 개발·운영 겸직 거부감 있나요? / 트래픽 대응은?
