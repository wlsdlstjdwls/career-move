2. 자기소개서 초안 (4문항)

실제 recruit.cj.net 로그인 후 노출되는 정확한 문항 번호·제목으로 맞춰야 함 — 아래는 "지원동기/직무역량/문제해결/포부" 통상 구조 초안

문항 1 · 지원동기
E-커머스 백엔드 엔지니어로 6년간 상품·주문·프로모션·클레임 도메인을 설계·개발·운영하며, 대규모 트래픽 환경에서 안정성과 성능을 동시에 확보하는 일에 강점을 쌓아왔습니다. 최근에는 더현대·투홈 플랫폼 통합 프로젝트에서 고객자산(예치금, H.Point, RSVP, 바우처) API를 직접 담당하며, 포인트·적립·바우처처럼 사용자 신뢰와 직결되는 도메인을 다루는 일의 무게를 체감했습니다. CJ ONE은 3천만 명 이상의 회원을 대상으로 한 통합 멤버십·포인트 시스템으로, 제가 다뤄온 "고객 자산" 도메인과 본질적으로 맞닿아 있습니다. 여러 브랜드·채널을 아우르는 대규모 트랜잭션을 안정적으로 처리하는 시스템을 만드는 일에 제 경험을 이어가고 싶어 지원했습니다.
문항 2 · 직무 역량
[대규모 트래픽·트랜잭션 처리 경험] 더현대·투홈 플랫폼 통합(NCP 구축)에서 주문상세·고객자산(예치금/H.Point/RSVP/바우처)·사은행사·선물받기 API를 담당했습니다. 오픈 직전 바우처 정책이 변경되며 옵션 포맷이 가변화되어 SUBSTR 기반 파싱 로직에서 오류가 발생했을 때, 영향받는 쿼리를 전수 조사하고 정규식 기반으로 일괄 교체한 뒤 유/무 케이스를 모두 검증해 서비스 중단 없이 반영했습니다. [성능 최적화 경험] PC몰 리뉴얼 프로젝트에서는 응답속도가 최대 4~5초까지 걸리던 주요 화면을 쿼리 튜닝과 Spring Cache 적용으로 1~2초대까지 단축했습니다. 카드즉시할인 프로모션 개발 시에는 TV·ARS·웹/앱 등 다중 채널에 걸친 영향 범위를 사전에 검증해 사이드이펙트 없이 안정적으로 오픈했습니다. [클라우드·모바일 서비스 개발 경험] 개인 프로젝트 'Fitness In Service'에서는 Spring Boot 멀티모듈(BO/MO/Batch) 구조로 회원 앱(MO) 전체를 직접 설계·개발했습니다. JWT 인증, Redis 세션, Kakao OAuth, Toss Payments 결제, SSE 실시간 알림까지 AWS EC2·Docker·Nginx 환경에 배포해 실제 서비스로 운영 중입니다. 기획부터 배포까지 전 과정을 책임진 경험을 통해 모바일 사용자 관점의 API 설계 감각을 키웠습니다.
문항 3 · 문제해결/협업 사례
기간계 시스템 고도화 프로젝트에서 Adobe Flex 기반 레거시 시스템을 ExtJS로 전환하는 작업에서 AA(Application Architect) 역할을 맡았습니다. 단순 기술 전환을 넘어 프로젝트 참여 인원을 대상으로 ExtJS 교육을 직접 진행하고, 코어·공통 모듈을 설계해 팀 전체가 일관된 방식으로 개발할 수 있는 기반을 만들었습니다. 오류·기술 문의 창구 역할도 겸하며 신규 시스템이 무중단으로 안정 운영되는 데 기여했습니다. 이 경험을 통해, 좋은 백엔드 개발자는 코드를 잘 짜는 것뿐 아니라 팀이 같은 기준으로 일할 수 있는 구조를 만드는 사람이라는 것을 배웠습니다.
문항 4 · 입사 후 포부
CJ ONE처럼 다수의 브랜드·채널이 얽힌 대규모 회원 시스템에서는 작은 정책 변경 하나가 여러 서비스에 파급되는 경우가 많다는 것을 이번 프로젝트들을 통해 체감했습니다. 입사 후에는 이런 변경에 안정적으로 대응할 수 있는 코드와 테스트 구조를 만드는 데 기여하고, 나아가 최근 관심을 두고 있는 LLM 기반 분석/자동화 경험(Claude, Gemini 등 멀티 AI 프로바이더 활용)을 백엔드 개발 생산성 향상에도 접목해보고 싶습니다.

성향 축 5개▸

대기업 인성검사는 대부분 "같은 성향을 다른 문항으로 여러 번 물어서 일관되는지" 보는 구조(사회적 바람직성 척도/라이 스케일 포함). "좋아 보이는 답"을 고르지 말고, 본인 성향을 몇 개 축으로 미리 정리해두고 그 축에서 안 벗어나게 답하는 게 전략.

본인 성향 축 정리
축본인 성향(근거)CFT 예상 문항
조직 충성도/이직 성향재입사 이력 있음 — "사람·문화는 만족, 처우·성장은 별개" 프레이밍 확정. 가장 예민하게 체크될 가능성 높은 축"소속감이 강한 편이다" / "더 나은 조건이면 언제든 이직" — 둘 다 극단 긍정하면 모순. "소속감은 있지만 처우·성장은 별개"톤 유지

코딩테스트 실전 대비

CJ 유형 리서치 포함▸
CJ올리브네트웍스 코딩테스트 정보 (2026.07.21 리서치)
직무별로 상이할 수 있음, 서류 통과 후 실제 안내 공지가 기준
  • 형식: 프로그래머스 플랫폼, 2시간에 2~3문제(Software Engineer 계열 후기는 2시간 3문제)
  • 난이도: 카카오·네이버보다 쉬운 편. 앞 1~2문제 프로그래머스 레벨1~2/백준 실버1~2급, 마지막 문제 골드4~5급
  • 언어: Java 지원 확인됨(당일 선택 가능 언어 4~5개로 제한된다는 후기 있으나 Java는 대부분 포함)
  • 함께 진행: 코딩테스트와 별도로 같은 플랫폼에서 인성검사도 진행 — 시간 여유 두고 응시

→ 우선순위: 문자열/배열·스택/큐 위주로 감 유지, 마지막 한 문제 대비로 BFS/DFS까지만 추가로 다지면 충분. DP·그리디는 우선순위 낮춰도 됨.

유형별 연습 순서 (전체 5개 유형, 우선순위 순)
  1. 문자열/배열 — 프로그래머스 "완주하지 못한 선수/위장/베스트앨범", 백준 1920/2941. HashMap을 언제 O(n)으로 쓰는지, 정렬이 왜 필요한지 구분해서 설명
  2. 스택/큐 — 백준 1874/2493/11866, 프로그래머스 "기능개발/프로세스". "왜 배열 대신 스택/큐인가"를 시간복잡도로 설명
  3. BFS/DFS — 백준 2178/11724/1260, 프로그래머스 "타겟 넘버/네트워크/여행경로". BFS=최단거리류, DFS=모든경로 탐색류 기준 먼저 말하기
  4. DP 기초 — 백준 1463/1932/2579/11726. 점화식을 먼저 쓰고 코드로 옮기는 순서 연습
  5. 그리디 — 백준 1931/11047/1541, 프로그래머스 "체육복/조이스틱". "왜 그리디로 풀리는가" 근거 설명 연습
문제 풀 때마다 하는 "설명 연습" 4단계
  1. 문제 재진술 — "이 문제는 결국 OO를 OO 조건 안에서 찾는 문제입니다"
  2. 접근 후보 비교 — 브루트포스부터 언급하고 최적화 근거 설명
  3. 선택한 자료구조/알고리즘과 이유 — 왜 그 도구를 골랐는지 한 문장
  4. 시간/공간복잡도 정리 — Big-O와 트레이드오프 언급

페이스: 주 3~4문제, 하루 30~40분. 1주차 문자열/배열 → 2주차 스택/큐 → 3~4주차 BFS/DFS → 5주차 DP → 6주차 그리디
