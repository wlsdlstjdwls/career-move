Design

설계 원칙

DDD 기반 멀티모듈

공유 도메인은 Core, 관리자(BO)·회원(MO)은 별도 모듈. 공유 코드는 반드시 Core에 두어 경계를 강제.

계층 & 경계 분리

Controller → Service → Mapper → Entity. DTO는 API 계층, VO는 Service↔Mapper — 도메인이 DTO에 의존하지 않게.

응답·예외 통일

모든 응답은 ApiResponse<T>, 예외는 BusinessException+에러코드 enum으로 일관되게 처리.

정합성·추적성 우선

상수 중앙화로 매직리터럴 제거, TraceId 필터+AOP로 요청 전 구간 로깅해 장애를 추적 가능하게.

Backend Design

백엔드 설계 포인트 — 배경 · 설계 · 이유

주어진 정책을 백엔드에서 정합성 있게 풀어낸 설계입니다.

상품 · 멤버십 · 수업 3계층 — 도메인/DB 설계

배경

회원이 실제 소유하는 건 상품이 아니라 차감·예약의 기준이 되는 권리(멤버십)이고, 개인+그룹이 섞인 패키지 상품도 팔아야 했습니다.

설계

판매 단위(Product)·소유/차감 단위(Membership)·운영 단위(Class)를 엔티티로 분리해 스키마에 반영(상품→멤버십 1:N, 멤버십→수업 N:1), 결제 성공 시 상품을 unpack해 멤버십을 N개 생성하도록 구현.

이유

패키지 상품을 단품 로직 변경 없이 확장할 수 있고, 화면별 표기 소스(스케줄=수업명, 보유함=멤버십명)를 분리해 혼선을 없앴습니다.

수업 취소·폐강의 차감 복구 정합성

배경

취소·폐강은 멤버십 차감 복구·정원·강사 급여에 연쇄로 영향을 주고, 취소 권한이 수업 유형·마감기한에 따라 달라지는 정책이 있었습니다.

설계

차감 복구는 트랜잭션으로 즉시 반영, 폐강은 배치 스케줄러가 최소 수강 인원 미달 시 자동 취소 → 전원 예약 취소 + 차감 복구 + 알림. 권한 정책은 수업 유형·마감 전후로 백엔드에서 분기 처리.

이유

돈·자원에 직결되는 상태(차감·정원)는 정합성을 최우선으로 두고, 권한은 운영 실수를 구조로 막도록 분기했습니다.

Problem Solving

문제 → 원인 → 해결 → 결과

운영하며 실제로 부딪힌 문제들. 코드 레벨 상세는 면접에서 설명 가능합니다.

결제가 이중으로 처리될 수 있는 문제 · 동시성·결제

문제

승인 요청과 Toss Webhook이 동시에 도착하거나 재시도가 겹치면 같은 결제가 두 번 반영될 수 있었습니다. 포인트·쿠폰이 섞인 혼합결제라 이중 차감/이중 환불 위험이 컸습니다.

원인

승인 로직과 Webhook 후처리가 각각 결제 완료 처리를 수행 → 동시 진입 시 중복 반영.

해결

승인과 Webhook이 동일한 비관적 락(SELECT … FOR UPDATE)으로 결제 행을 잠그고 PG 이력이 0건일 때만 후처리. 승인은 멱등 처리(이미 완료면 즉시 성공 반환)+Toss 멱등키. 포인트 잔액도 FOR UPDATE로 잠그고, 환불은 잔여 환불가능액 기준으로 처리 후 즉시 갱신.

결과

동시 요청·재시도에도 결제 1건당 정확히 1회 반영. 실패 이력은 REQUIRES_NEW로 분리해 승인 롤백과 무관하게 보존.

목록 조회가 데이터에 비례해 느려지는 문제 · 성능·SQL

문제

결제·회원권 목록에서 환불가능액·잔여 세션 수를 계산하는 부분이 데이터가 늘수록 급격히 느려졌습니다.

원인

계산을 행마다 상관 서브쿼리로 수행 → 행 수만큼 쿼리가 반복 실행(N+1).

해결

상관 서브쿼리를 LEFT JOIN 집계로 재작성, 무제한 목록에 페이지네이션/안전 LIMIT, 조회·조인 패턴에 맞춘 복합 인덱스 설계.

결과

목록 조회가 데이터량과 무관하게 상수 개 쿼리로 동작.

알림·이력이 트랜잭션에 잘못 묶이는 문제 · 트랜잭션·배치

문제

알림 발송이나 실패 이력이 본 트랜잭션에 묶이면, 롤백 시 이력이 사라지거나 커밋 전에 알림이 먼저 나갔습니다. 배치에서는 한 건 오류가 전체를 롤백시켰습니다.

원인

부가 작업(알림·이력)과 핵심 처리가 같은 트랜잭션 경계 안에 있었음.

해결

실패 이력·주문 만료 배치는 REQUIRES_NEW로 건별 독립 트랜잭션, 알림은 @Async+커밋 후 콜백(afterCommit)으로 "커밋 성공 이후에만" 발송.

결과

부분 실패가 전체를 되돌리지 않고, 롤백된 거래의 알림이 잘못 나가는 일이 사라짐.

급여·잔여세션이 잘못 집계되던 버그 · 데이터 정합성

문제

"예약중" 세션이 "소진된 세션"으로 잘못 집계되어 잔여 세션·급여 산정에 오차가 생겼습니다.

원인

상태 문자열 오타("RESERVE" vs 실제 "RESERVED")로 필터가 항상 참이 됐고, 같은 오타가 복사되며 다른 모듈로 전파됨.

해결

고치기 전 급여정산의 화이트리스트(COMPLETED/NO_SHOW만 인정)와 대조해 의도를 검증하고, 오타 전파 여부를 전수 검색으로 확인 후 수정.

결과

집계 정확도 회복. "정상 범위를 좁게 정의하는 화이트리스트가 실수에 강하다"를 기준으로 확립.

도메인 구분 한계 → 패키지 구조로 확장 · 확장성·운영안정

문제

수업을 class_type으로 도메인만 구분해, 같은 도메인 내 개인/그룹/듀엣 형식을 나눌 수 없었고 개인+그룹 혼합 패키지도 팔 수 없었습니다.

원인

판매(상품)·운영(수업) 단위가 한 축(class_type)에 묶여 있어 확장이 막힘.

해결

product_type='PACKAGE'+구성항목 테이블을 신설해 결제 시 unpack하여 멤버십 N개 생성. 기존 상품 목록 쿼리는 status·display 조건이라 패키지가 자동 포함 → 구매 플로우만 분기하고 나머지는 무변경.

결과

단품 로직에 영향 없이 패키지 판매 확장. "운영 중 자산은 건드리지 않고 확장은 격리"라는 원칙과 일치.
