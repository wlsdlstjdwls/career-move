## 1. 자격요건 대조 (지원 전 자가 점검)

### 지원자격 (필수)

| 요건 | 본인 현황 | 대응 |
|------|-----------|------|
| 백엔드 개발 경력 5~10년 | 6년차, 정확히 부합 | 그대로 어필 |
| Java 기반 서비스 개발 | Java 6년+ | 강점 |
| Spring/Spring Boot 기반 개발 | Spring 6년+, Fitness In Service는 Spring Boot 4.0 | 강점 |
| RESTful API 설계·개발 | 주문상세/고객자산/사은행사/선물받기 API 등 다수 | 강점 |
| RDBMS(MySQL/Oracle/PostgreSQL) 활용 | Oracle·PL/SQL 5년+, MariaDB 사용 | Oracle은 정확히 일치, MySQL/PostgreSQL은 실무 경험 명시 어려움 — "Oracle 기반 대규모 쿼리 튜닝 경험, RDBMS 전반 원리는 동일하게 적용 가능" 식으로 방어 |
| 대규모 서비스 설계·구축·운영 경험 | 더현대·투홈 NCP 통합(현대백화점그룹 대형 플랫폼) | 강점, 단 "자사 서비스"가 아니라 "고객사(현대백화점그룹) 대형 플랫폼을 벤더사 소속으로 설계·운영"한 것임을 정직하게 기술 |
| **Kubernetes를 활용한 서비스 개발** | **경험 없음/제한적** (Docker·Nginx까지는 있음, K8s는 career-description에 없음) | **가장 큰 갭.** 허위 기재 금지. 자소서/면접에서 "Docker 기반 컨테이너 운영 경험은 있으나 K8s 실무 경험은 없음, 개념 학습 중"으로 정직하게 밝히고 학습 계획 제시 |

### Skill/Knowledge — 우대사항 중 본인이 방어 가능한 항목

| 항목 | 본인 현황 |
|------|-----------|
| OAuth2.1/JWT/Session/Cookie 인증 이해 | JWT Access/Refresh 토큰 구조 직접 설계(Fitness In Service), Redis 세션 관리 — **강점** |
| Redis 활용 | Fitness In Service에서 세션 관리에 사용 — 있음. 단 "Redis Cluster 운영 경험"(우대사항)은 없음 — 과장 금지 |
| DB 모델링/SQL 최적화 | PC몰 쿼리 튜닝(3~5초→1~2초), Oracle PL/SQL 튜닝 경험 — **강점** |
| CI/CD·DevOps 이해 | Jenkins 기반 CI/CD 구축 경험(기간계 시스템 고도화) — 있음. 단 공고 기술스택은 Gitlab/ArgoCd — 도구는 다르지만 "CI/CD 파이프라인 설계 원리"로 방어 |
| MSA/분산시스템 이해 | **경험 없음** — Fitness In Service는 멀티모듈 모놀리식 구조, MSA 아님. 이 부분은 우대사항이라 필수 탈락 사유는 아니지만 정직하게 인정 필요 |
| OAuth2 Authorization Server 구축, OIDC, SSO | **경험 없음** — Kakao OAuth는 소셜로그인 "클라이언트" 연동이지 "Authorization Server 구축"이 아님. 우대사항이므로 무리하게 있다고 쓰지 말 것 |
| 대규모 트래픽(수백만 회원 이상) 운영 | 더현대·투홈 통합 규모가 수백만 회원급인지는 본인이 정확한 수치를 모름 — 자소서에 구체 숫자 기재 금지, "대형 유통그룹 통합 플랫폼" 정도로 표현 |
| 기술 리딩/프로젝트 리딩 | 기간계 시스템 고도화에서 AA 역할, ExtJS 교육 진행 — **강점** |

**결론**: 필수요건(경력연차·Java/Spring·RESTful·RDBMS·대규모 서비스)은 충족. **Kubernetes 실무 경험 부재**가 유일한 필수요건 갭이고, 우대사항 중 인증서버(OAuth2 AS/OIDC/SSO)·MSA·Redis Cluster는 미보유. 지원은 권장하되, 자소서·면접에서 이 갭을 정직하게 인정하고 "무엇으로 메꿀 것인지"(학습 계획, 전이 가능한 경험)를 명확히 제시하는 방향으로 작성.

## 2. 자기소개서 초안 (일반적인 CJ 경력직 4문항 구조 기준)

> 주의: recruit.cj.net에서 실제 지원서 작성 시 노출되는 정확한 문항으로 다시 맞춰야 함. 아래는 통상적인 "지원동기 / 직무역량 / 문제해결 사례 / 입사 후 포부" 구조로 작성한 초안이라 문항 번호·제목만 실제 공고에 맞게 조정하면 됨.

### 문항 1. 지원동기 — CJ ONE 백엔드 개발자에 지원한 이유

```
E-커머스 백엔드 엔지니어로 6년간 상품·주문·프로모션·클레임 도메인을 설계·개발·운영하며, 대규모 트래픽 환경에서 안정성과 성능을 동시에 확보하는 일에 강점을 쌓아왔습니다. 최근에는 더현대·투홈 플랫폼 통합 프로젝트에서 고객자산(예치금, H.Point, RSVP, 바우처) API를 직접 담당하며, 포인트·적립·바우처처럼 사용자 신뢰와 직결되는 도메인을 다루는 일의 무게를 체감했습니다.

CJ ONE은 3천만 명 이상의 회원을 대상으로 한 통합 멤버십·포인트 시스템으로, 제가 다뤄온 "고객 자산" 도메인과 본질적으로 맞닿아 있습니다. 여러 브랜드·채널을 아우르는 대규모 트랜잭션을 안정적으로 처리하는 시스템을 만드는 일에 제 경험을 이어가고 싶어 지원했습니다.
```

### 문항 2. 직무 역량 — 핵심 경험 기술

```
[대규모 트래픽·트랜잭션 처리 경험]
더현대·투홈 플랫폼 통합(NCP 구축)에서 주문상세·고객자산(예치금/H.Point/RSVP/바우처)·사은행사·선물받기 API를 담당했습니다. 오픈 직전 바우처 정책이 변경되며 옵션 포맷이 가변화되어 SUBSTR 기반 파싱 로직에서 오류가 발생했을 때, 영향받는 쿼리를 전수 조사하고 정규식 기반으로 일괄 교체한 뒤 유/무 케이스를 모두 검증해 서비스 중단 없이 반영했습니다.

[성능 최적화 경험]
PC몰 리뉴얼 프로젝트에서는 응답속도가 최대 4~5초까지 걸리던 주요 화면을 쿼리 튜닝과 Spring Cache 적용으로 1~2초대까지 단축했습니다. 카드즉시할인 프로모션 개발 시에는 TV·ARS·웹/앱 등 다중 채널에 걸친 영향 범위를 사전에 검증해 사이드이펙트 없이 안정적으로 오픈했습니다.

[클라우드·모바일 서비스 개발 경험]
개인 프로젝트 'Fitness In Service'에서는 Spring Boot 멀티모듈(BO/MO/Batch) 구조로 회원 앱(MO) 전체를 직접 설계·개발했습니다. JWT 인증, Redis 세션, Kakao OAuth, Toss Payments 결제, SSE 실시간 알림까지 AWS EC2·Docker·Nginx 환경에 배포해 실제 서비스로 운영 중입니다. 기획부터 배포까지 전 과정을 책임진 경험을 통해 모바일 사용자 관점의 API 설계 감각을 키웠습니다.
```

### 문항 3. 문제해결/협업 사례

```
(위 "바우처 정책 변경 대응" 사례를 이 문항 전용으로 더 길게 풀어쓰거나, 아래 사례로 대체 가능)

기간계 시스템 고도화 프로젝트에서 Adobe Flex 기반 레거시 시스템을 ExtJS로 전환하는 작업에서 AA(Application Architect) 역할을 맡았습니다. 단순 기술 전환을 넘어 프로젝트 참여 인원을 대상으로 ExtJS 교육을 직접 진행하고, 코어·공통 모듈을 설계해 팀 전체가 일관된 방식으로 개발할 수 있는 기반을 만들었습니다. 오류·기술 문의 창구 역할도 겸하며 신규 시스템이 무중단으로 안정 운영되는 데 기여했습니다.

이 경험을 통해, 좋은 백엔드 개발자는 코드를 잘 짜는 것뿐 아니라 팀이 같은 기준으로 일할 수 있는 구조를 만드는 사람이라는 것을 배웠습니다.
```
