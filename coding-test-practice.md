# 코딩테스트 실전 연습 리스트

`study-plan.md` 섹션 1("코딩테스트")의 실행판. 6년차 기준 목표는 "다 풀기"가 아니라 "정해진 시간 안에 정확한 풀이를 고르고, 왜 그 자료구조/알고리즘을 골랐는지 말로 설명하는 것" — 아래 문제는 유형별 대표 예시이며, 실제 풀 때는 플랫폼에서 최신 난이도 태그를 다시 확인할 것(문제 난이도는 시간이 지나며 재조정되기도 함).

## 유형별 연습 순서 (5개 유형, 우선순위 순)

### 1. 문자열/배열 (기초 감각 유지)
- 프로그래머스: "완주하지 못한 선수" (해시), "위장" (해시), "베스트앨범" (해시+정렬)
- 백준: 1920(수 찾기, 이분탐색), 2941(크로아티아 알파벳, 문자열 처리)
- 연습 포인트: HashMap을 언제 O(n)으로 쓰는지, 정렬이 왜 필요한 시점인지 구분해서 설명

### 2. 스택/큐
- 백준: 1874(스택 수열), 2493(탑, 스택 활용), 11866(요세푸스 문제, 큐)
- 프로그래머스: "기능개발" (큐), "프로세스" (큐+우선순위)
- 연습 포인트: "왜 배열 대신 스택/큐를 골랐는가"를 시간복잡도 근거로 설명(예: 배열에서 앞쪽 삭제는 O(n), 큐는 O(1))

### 3. BFS/DFS
- 백준: 2178(미로 탐색, BFS), 11724(연결 요소의 개수, DFS/BFS), 1260(DFS와 BFS)
- 프로그래머스: "타겟 넘버" (DFS), "네트워크" (DFS/BFS), "여행경로" (DFS+백트래킹)
- 연습 포인트: BFS는 "최단 거리류", DFS는 "모든 경로 탐색/조합류"라는 선택 기준을 먼저 말하고 코드로 들어가는 습관

### 4. DP 기초
- 백준: 1463(1로 만들기), 1932(정수 삼각형), 2579(계단 오르기), 11726(2×n 타일링)
- 프로그래머스: "N으로 표현", "정수 삼각형"
- 연습 포인트: 점화식을 화이트보드에 먼저 쓰고 코드로 옮기는 순서 연습 — 면접관 앞에서는 코드보다 점화식 도출 과정을 보여주는 게 더 중요

### 5. 그리디
- 백준: 1931(회의실 배정), 11047(동전 0), 1541(잃어버린 괄호)
- 프로그래머스: "체육복", "조이스틱"
- 연습 포인트: "이 문제가 왜 그리디로 풀리는가"(지역 최적해가 전역 최적해로 이어지는 근거)를 설명하는 연습 — 이 논리를 못 대면 실제로는 DP가 필요한 문제를 그리디로 잘못 푼 것일 수 있음

## 주 단위 페이스 (study-plan.md와 동일 기준)
- 주 3~4문제, 하루 30~40분
- 순서: 1주차 문자열/배열 → 2주차 스택/큐 → 3~4주차 BFS/DFS → 5주차 DP → 6주차 그리디, 이후 반복 복습
- 코딩테스트가 임박한 회사(예: CJ올리브네트웍스)가 있으면 해당 회사 후기(잡플래닛/블라인드)에서 실제 출제 유형을 먼저 확인하고 그 유형 비중을 높일 것

## 문제 풀 때마다 하는 "설명 연습" 템플릿

문제를 풀고 나서, 아래 4단계를 소리 내어 말하는 것까지가 한 세트. 코드만 통과시키고 넘어가면 코딩테스트 감은 늘어도 면접 설명력은 안 늘어남.

1. **문제 재진술**: "이 문제는 결국 OO를 OO 조건 안에서 찾는 문제입니다" — 면접관이 문제를 정확히 이해했는지 확인하는 단계이기도 함
2. **접근 후보 비교**: "완전탐색으로 하면 O(n²)인데, 제약조건상 n이 커서 시간초과 날 수 있어 O(n log n)으로 줄일 방법을 생각했습니다" — 브루트포스부터 언급하고 최적화 근거를 설명하는 순서가 면접에서 좋은 인상을 줌
3. **선택한 자료구조/알고리즘과 이유**: "그래서 해시맵을 써서 조회를 O(1)로 줄였습니다" 식으로 왜 그 도구를 골랐는지 한 문장
4. **시간/공간복잡도 정리**: 최종 풀이의 Big-O를 말하고, 트레이드오프(공간을 더 써서 시간을 줄였는지 등)가 있으면 언급
