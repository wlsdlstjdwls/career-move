Q. CJ 기업 3가지 특성(정직/열정/창의) 중 본인과 가장 잘 맞는 것은?

"정직"이 답변축 — Kubernetes 경험 없다고 있는 그대로 말씀드린 것처럼, 모르는 것과 아는 것을 정확히 구분해 말하고 문제를 축소하지 않는 태도가 실무에서도 마찬가지라는 논리. 2026.07.22 업데이트: "하고잡이/We Who Prove It"(선언에 그치지 않고 결과로 증명)와 자연스럽게 연결 — "정직이 전제, 그 위에서 하고잡이는 끝까지 실행해 결과로 증명하는 것"으로 위계 연결. "하고잡이" 단어는 1회만 언급.

Q. (예상) DX·AX가 본인 직무와 어떻게 연결된다고 생각하나요?

"CJ ONE 슈퍼앱 진화"(뉴비전 3대 과제)를 대규모 회원 데이터의 AI 초개인화 방향으로 이해, 그 전제가 되는 "안정적 데이터 적재·정합성"이 본인 역할이라는 답변. 회사가 DX/AX를 명시적으로 구분 정의한 문서는 없으므로 단정적으로 말하지 말 것 — "제가 이해하기로는"의 톤 유지.

Q. 무리한 일정의 지시가 내려온다면?

"무조건 맞춘다"도 "무조건 반발한다"도 아닌, 영향 범위를 먼저 전수 조사해 "여기까지는 되고 여기는 리스크가 있다"를 구체적으로 공유하고 우선순위를 같이 정하는 방식 — 바우처 정책 변경 대응 사례 재사용.

Q. 앞으로 커리어 방향 / 개발문화 형성 방안 / 인생관은?

커리어: 안정성·정합성 중심 백엔드 도메인 심화 + AI 에이전트 활용 결합. 개발문화: "모르는 걸 숨기지 않는 문화"가 최우선 — 바우처 대응 때 문제를 축소 않고 전수 조사해 공유한 태도를 팀 단위로 확산. 인생관은 본인 실제 가치관으로 채워야 하는 항목(미완성).

Q. 본인이 개발한 것 중 CJ ONE에 적용 가능한 게 뭐가 있나요?

고객자산(예치금/H.Point/바우처) API 경험 + Fitness In Service의 JWT Access/Refresh·Redis 세션 설계 경험을 CJ ONE의 회원·포인트·인증 도메인에 1:1로 매칭.

Q. CJ ONE의 최근 사업 방향 / 최근 관심 있게 본 뉴스는?

이마트24·야놀자 등 비계열 제휴 확대("개방형 멤버십")와 CJ PAY 우리통장(우리은행 제휴 핀테크 상품)을 2025년 이후 최신 동향으로 언급 — 제휴사가 늘수록 회원 인증·계정 관리가 단순 로그인을 넘어 여러 파트너사가 신뢰하는 인프라가 돼야 한다는 논리로 지원 이유와 연결. 출처 기사(파이낸셜뉴스 3100만 기사, CJ PAY 우리통장 머니투데이 기사)는 면접 전 한 번 더 확인.

Q. (예상) 회원서비스 직무를 맡으면 구체적으로 어떤 업무를 하게 될 것 같나요?

회원가입/탈퇴/재가입, 소셜로그인 연동, 휴면계정 전환, 개인정보 동의관리, 세션·토큰 인증체계 + 개방형 멤버십 특성상 제휴사 계정·포인트 매핑까지 추정. 확정 정보 아님 — "제 나름대로 추정해본 것"이라는 톤 유지, 단정하면 오히려 아는 척하는 인상.

Q. CJ ONE이 Kotlin/Ktor 쓰는 팀도 있다던데 아시나요? / Kotlin 써보셨나요?

실무 경험 없음을 정직히 인정 + JVM 기반이라 Java 경험으로 빠르게 전환 가능하다는 논리(K8s 답변과 동일 톤). Ktor는 Spring보다 가볍고 코루틴 기반 비동기 중심이라는 것만 개념적으로 파악.

Q. 고객사 요구사항 vs 회사 기술력, 하나 고른다면? / 무리한 고객사 요구사항 해결법은?

무조건 한쪽을 고르기보다, 요구사항을 그대로 수용했을 때의 리스크(안정성·유지보수성)를 구체적으로 설명해 고객사가 판단할 정보를 주고 안전한 대안을 제시 — 더현대·투홈 바우처 정책 변경 대응이 실제 사례.

Q. 서버 메모리 튜닝 과정은? / 개발·운영 겸직 거부감 있나요? / 트래픽 대응은?

주의: 이 저장소 경력 기록엔 JVM 메모리 튜닝 실무 경험이 명시돼 있지 않음 — 힙/GC 트레이드오프 개념은 이해하되 실무 경험은 없다고 정직히 구분. 겸직은 "전혀 거부감 없다"(Fitness In Service를 AWS EC2/Docker/Nginx로 직접 배포·운영한 경험). 트래픽 대응은 쿼리 튜닝·Spring Cache·Redis 경험 + 레플리케이션/샤딩 방향 검토.

Q. 개발환경(로컬~운영) 관련 질문 대비

⚠️ 잡플래닛 후기 중 유일하게 탈락 원인이 명시된 질문 유형("개발환경 관련 대답을 잘 하지 못해 탈락한 듯 함", 2023-07 개발|대리) — Jenkins CI/CD + Gradle 멀티모듈 + Docker/Nginx 배포 경험으로 답변, Gitlab/ArgoCD 같은 CJ 공고의 구체 도구는 안 써봤지만 원리는 이해한다고 구분해 답할 것.

Q. 신기술이 아니라 레거시만 접하게 된다면 어떨 것 같나요?

실제 합격 사례(2022-04) 원문 질문. 본인 스택(MyBatis·Oracle PL/SQL, Adobe Flex→ExtJS 전환)이 실제로 레거시 위주라 정직하게 답하기 좋은 소재 — "어렵고 시간 더 걸리는 건 사실이지만, 레거시를 안전하게 유지보수하는 능력이 신기술을 다룰 때도 도움된다"는 논리.

Q. 잦은 이직사유는 무엇인가요?

Q. 입사 후 목표는 무엇인가요?

입사 후 1년 안에는 CJ ONE의 회원·인증 도메인을 실무 코드 레벨까지 완전히 파악하고, 갭으로 인정한 Kubernetes 운영 경험을 실무에서 실제로 쌓는 걸 첫 목표로 삼고 있습니다. 이후엔 더현대·투홈 통합에서처럼 정책 변경·장애 상황에서 영향 범위를 빠르게 파악하고 대응하는 역할까지 올라가고, 장기적으로는 회원 시스템 설계 결정에 직접 참여하는 포지션으로 성장하고 싶습니다.

짧은 버전: 1년 내 CJ ONE 도메인 완전 파악 + K8s 실무 경험 확보, 이후 정책 변경·장애 대응을 주도할 수 있는 역할로 성장. · 꼬리질문 "목표 달성 여부 확인법?" → K8s 실무 클러스터 운영 여부, 도움 없이 장애 원인 파악 가능 여부로 자가 점검.

Q. 실수를 했을 때 어떻게 대응했나요?

실수를 발견하면 숨기지 않고 바로 알리는 게 우선입니다. (확인 필요: 실제 겪은 실수 사례) 발견 즉시 영향 범위를 파악해 팀에 공유하고, 되돌릴 수 있는 부분은 롤백·핫픽스로 대응했습니다. 이후엔 재발 방지를 위해 (확인 필요: 체크리스트/테스트 보강 등)로 프로세스를 조정했습니다. 실수 자체보다 얼마나 빨리 알리고 얼마나 정직하게 영향 범위를 밝히는지가 신뢰를 좌우한다고 생각합니다.

⚠️ 본인 실제 사례로 채워야 완성 — 없으면 "실패했거나 아쉬웠던 프로젝트" 소재 재사용 가능. 꼬리질문 "숨기고 싶었던 적은?" → 늦게 알려질수록 영향 범위가 커진다는 논리로 정직 축 유지.

Q. 본인이 팀에 기여할 수 있는 점은 무엇인가요?

두 가지입니다. 첫째는 변경의 영향 범위를 끝까지 추적하는 꼼꼼함 — 바우처 정책 변경 대응처럼 전수 조사 후 배포하는 습관이 팀 전체 배포 안정성에 도움됩니다. 둘째는 모르는 것을 숨기지 않는 태도 — K8s 경험 없음을 그대로 밝힌 것처럼, 팀 내에서도 모르는 부분을 빨리 드러내야 서로 도울 수 있다고 믿습니다.

짧은 버전: 영향 범위를 끝까지 추적하는 꼼꼼함 + 모르는 걸 숨기지 않는 정직한 커뮤니케이션. ⚠️ "이 직무에 필요한 역량" 답변과 구조 유사 — 연달아 나오면 표현 조정 필요.

Q. 대학 시절 의미 있었던 활동이나 프로젝트는? 참고용

6년차 경력직 면접엔 잘 안 나오는 신입형 질문. (확인 필요: 실제 대학 활동 한 가지) 짧게 언급 후, 곧바로 6년간 쌓은 실무 경험(더현대·투홈 통합 등)으로 자연스럽게 전환하는 흐름 자체가 경력직다운 답변으로 보임.

Q. 지원 직무와 전공·경험이 다르면 어떻게 적응할 것인가요? 참고용

이 문구 그대로보다 "기술 스택 갭"(K8s·Kotlin)으로 변형되어 나올 가능성이 큼. 가장 가까운 기존 경험(Docker 배포·운영)에서 원리를 확장해 개념을 빠르게 학습하고, 실무 투입 후 검증하며 채워나가는 방식으로 적응한다고 답변.

꼬리질문 "그래도 적응 안 되면?" → 숨기지 않고 팀에 도움 요청 — 정직 축과 통일.

Q. 팀보다 개인의 성과가 우선이라는 상황에서 어떻게 행동하겠는가?

개인 성과가 팀 전체 결과에 어떤 영향을 주는지부터 따집니다. 기간계 시스템 고도화에서 AA 역할을 맡았을 때도 개인 구현 속도보다 팀 전체가 같은 기준으로 개발할 수 있도록 공통 모듈 설계·교육에 더 시간을 썼는데, 결과적으로 그게 더 큰 기여였습니다. 개인 성과를 무시하진 않지만 판단 기준은 항상 "팀 전체 결과에 어떤 영향을 주는가"입니다.

꼬리질문 "개인 평가에서 손해 봐도 괜찮나?" → 팀 결과가 좋아지면 그 안에서 기여도 같이 드러난다는 논리로 방어.

Q. 지원 직무 관련 현재 개선이 필요하다고 생각하는 기술·역량은?

Kubernetes 실무 운영 역량이 가장 먼저입니다 — Docker 배포·운영 경험은 있지만 K8s는 개념 학습 단계라, Pod/Service/Deployment부터 학습 중이고 입사 후 실무 클러스터로 채우고 싶습니다. 두 번째는 MSA·분산 시스템 설계 — 지금까지는 멀티모듈 모놀리식까지만 다뤄봤습니다.

⚠️ K8s 갭 답변과 소재 겹침 — 같이 나오면 MSA 비중을 더 실어 반복처럼 안 들리게 조정.

⚠️ 완전 개인 질문, 본인 사례로 채워야 완성. 후보: 기간계 고도화 중 팀원에게 받은 코드 리뷰 코멘트.

Q. 자신 있는 프로그래밍 언어와 그 언어로 만든 프로젝트는?

Java입니다. 6년+ Spring/Spring Boot 실무 개발, 최근 개인 프로젝트 Fitness In Service를 Spring Boot 4.0 멀티모듈(core/bo-api/mo-api/batch)로 설계·개발·배포까지 혼자 진행 — JWT 인증, Redis 세션, Kakao OAuth, Toss 결제, SSE 실시간 알림 구현 후 AWS EC2·Docker·Nginx에 배포·운영 중. 실무에서는 더현대·투홈 고객자산 API를 Java/Spring으로 단독 개발.

꼬리질문 "Java 17/21 신기능 실무 사용 경험?" → (확인 필요), 없으면 학습만 했다고 정직하게.

Q. 객체지향 설계 원칙(SOLID)을 적용한 사례가 있다면?

기간계 시스템 고도화에서 AA 역할로 코어/공통 모듈 설계 시 SRP·OCP를 의식적으로 적용 — 공통 로직을 중복 없이 분리하고 신규 요구사항에도 기존 코드 수정 없이 확장 가능한 구조를 목표로 함. Fitness In Service core 모듈도 단방향 의존으로 ISP·DIP를 실무 규모에서 적용.

꼬리질문 "OCP 지키려다 과설계된 적은?" → 확장 포인트 과설계 경험 인정 + 이후 "지금 필요한 확장만" 원칙으로 조정한 실용주의적 답변.

Q. 코드 리뷰를 받았을 때 어떻게 대처하시나요?

먼저 방어하기보다 코멘트 의도부터 파악 — 놓친 케이스면 바로 인정·반영, 의도가 있던 부분이면 근거를 남겨 재논의. 기간계 고도화에서 팀에 ExtJS 표준을 교육했던 경험도, 코드 리뷰가 지적이 아니라 팀 기준을 맞춰가는 과정이라고 생각하게 된 계기였습니다.
